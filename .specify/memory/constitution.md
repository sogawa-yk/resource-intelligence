# Resource Intelligence Constitution

## Core Principles

### I. 日本語ドキュメント必須

本プロジェクトにおけるすべてのドキュメントは日本語で作成しなければならない（MUST）。
対象には仕様書、設計書、タスク一覧、コミットメッセージの本文、PRの説明文、
コード内のドキュメンテーションコメントを含む。

- コード内の変数名・関数名・識別子は英語を使用する（MUST）
- インラインコメントは日本語を推奨するが、英語も許容する
- 外部ライブラリのドキュメント引用等、原文が英語の場合はそのまま引用してよい

### II. 包括的テスト戦略（非交渉）

実装が完了したすべての機能に対して、以下の4種類のテストを
必ず実施しなければならない（MUST）：

1. **単体テスト**: 個々の関数・メソッドの正確性を検証する
2. **結合テスト**: コンポーネント間の連携・契約を検証する
3. **パフォーマンステスト**: レスポンスタイム、スループット、
   リソース消費量が許容範囲内であることを検証する
4. **E2Eテスト**: ユーザーシナリオに基づくエンドツーエンドの
   動作を検証する

- テストなしのマージは禁止する（MUST NOT）
- テストは実装前に設計する（テストファースト推奨）

### III. Kubernetes環境テスト（非交渉）

すべてのテストはKubernetes環境にデプロイした状態で
実施しなければならない（MUST）。ローカル環境のみでの
テスト実行は最終的な検証として認めない。

- 開発中のローカルテスト実行は許容するが、マージ前に
  Kubernetes環境での検証が必須である（MUST）
- CI/CDパイプラインはKubernetesクラスタへのデプロイと
  テスト実行を自動化しなければならない（MUST）
- テスト用のKubernetes名前空間を機能ブランチごとに
  分離することを推奨する

### IV. 可観測性

本番環境およびテスト環境において、システムの動作を
把握できる仕組みを備えなければならない（MUST）。

- 構造化ログを使用する（MUST）
- エラー発生時にはトレースIDを含むログを出力する（MUST）
- メトリクスのエクスポートを実装する（SHOULD）

### V. シンプルさ

必要最小限の複雑さで実装する。YAGNIの原則に従い、
現時点で必要のない機能は実装しない。

- 抽象化は具体的な必要性が2回以上確認されてから導入する
- 依存関係は最小限に保つ（MUST）
- 複雑さの追加には正当な理由をドキュメントに記載する（MUST）

## テスト要件

各テストレベルの具体的な要件を以下に定める。

### 単体テスト

- カバレッジ目標: 主要ロジックの80%以上
- モック・スタブの使用を許容する
- 実行時間: 個別テスト1秒以内を目標とする

### 結合テスト

- OCI APIとの連携をテストする
- データベース（使用する場合）との連携をテストする
- コンポーネント間のインターフェース契約を検証する

### パフォーマンステスト

- レスポンスタイムの閾値を事前に定義する（MUST）
- 並行リクエスト処理能力を検証する
- リソース消費量（CPU・メモリ）を計測する

### E2Eテスト

- ユーザーストーリーごとにE2Eシナリオを作成する（MUST）
- Kubernetes環境上でのエンドツーエンド動作を検証する
- 異常系シナリオも含める（MUST）

## 開発ワークフロー

### コードレビュー

- すべてのPRはレビューを経てからマージする（MUST）
- Constitutionへの準拠をレビュー観点に含める（MUST）

### デプロイメント

- 機能ブランチごとにKubernetes名前空間を作成する
- テスト通過後のみマージを許可する（MUST）
- デプロイはCI/CDパイプラインを通じて実行する（MUST）

### 品質ゲート

1. コードレビュー承認
2. 単体テスト通過（Kubernetes環境）
3. 結合テスト通過（Kubernetes環境）
4. パフォーマンステスト通過（Kubernetes環境）
5. E2Eテスト通過（Kubernetes環境）

## Governance

本Constitutionはプロジェクトのすべての開発プラクティスに
優先する。

- 修正にはドキュメント化された提案、承認、移行計画が必要である
- すべてのPR・レビューはConstitution準拠を検証しなければならない
- バージョニングはセマンティックバージョニングに従う：
  - MAJOR: 原則の削除・再定義等の後方互換性のない変更
  - MINOR: 新原則・セクションの追加、既存内容の大幅拡充
  - PATCH: 表現の修正、誤字脱字の修正、非意味的な改善
